# 04 · 일정 & 위험 관리 (Schedule & Risks)

> **기간 정리**: 6주 일정 = **10주차 ~ 15주차**. (12주차 **중간 발표**, 15주차 **최종 발표** 반영 시 6주가 정확히 맞아, 종료 주차를 15주차로 둠.)
> 전략: **수직 슬라이스 우선** — 12주차 중간 발표 시점에 *등록→OCR→태그→필터→로컬저장*의 Must 흐름이 end-to-end로 동작하는 것을 1차 목표로 한다.

---

## 주차별 일정

### 10주차 — 설계 확정 & 프로젝트 세팅
- **목표**: 만들 것을 못 박는다. 데이터 모델·화면 흐름·아키텍처 결정, Flutter 프로젝트 골격 세팅.
- **산출물**: 02-wbs 기반 설계 문서(엔티티 스키마, 화면 흐름도, 와이어프레임), 빈 화면으로 라우팅 되는 Flutter 스캐폴드, 로컬 DB(Isar/Drift) 연결.
- **검증**: 빈 앱이 빌드·실행되고 화면 간 이동이 됨 / 더미 레코드 1건 로컬 DB 저장·조회 성공.
- **WBS**: 1.1, 1.2, 2.1, 2.2, 2.3

### 11주차 — 입력 & OCR MVP
- **목표**: 사진 한 장이 텍스트가 되는 핵심 파이프라인 완성.
- **산출물**: 카메라/갤러리 입력 → ML Kit OCR → 추출 텍스트 표시·편집 화면.
- **검증**: 실제 교재 사진 5장으로 한/영 텍스트 추출 확인, 편집 후 값 유지 / 비행기 모드에서 OCR 동작.
- **WBS**: 3.1, 3.2.1~3.2.2, 3.3, 5.3.2

### 12주차 — 분류·저장 핵심 + 🎤 중간 발표
- **목표**: Must 흐름을 end-to-end로 연결해 **발표 가능한 데모** 완성.
- **산출물**: 태그 부착, 로컬 저장(이미지+텍스트+태그), 목록/상세, 태그 필터. 중간 발표 자료.
- **검증**: "촬영→OCR→태그→저장→태그로 필터→상세 열람" 시연 무중단 통과 / 앱 재실행 후 데이터 보존.
- **WBS**: 4.1.1, 4.2, 4.3.1, 1.3.1

### 13주차 — 검색·필터 완성 + 내보내기/가져오기
- **목표**: "다시 찾기"와 "데이터 주권" 가치(Should) 구현.
- **산출물**: OCR 텍스트 키워드 검색, 정렬, 내보내기/가져오기(zip), 복습 상태.
- **검증**: 키워드로 해당 오답 검색됨 / 내보낸 파일을 초기화 후 가져와 round-trip 일치 / 정렬 정상.
- **WBS**: 4.3.2~4.3.3, 5.1, 5.2.1

### 14주차 — 부가기능 일부 + 안정화·테스트
- **목표**: Could 일부 반영, 품질 끌어올리기.
- **산출물**: 태그 자동 추천, 이미지 전처리/다크모드 중 우선순위 1~2개, 단위/위젯 테스트, OCR 정확도 측정 리포트.
- **검증**: 대량 데이터(100건+)에서 목록·검색 성능 허용범위 / OCR 정확도 측정셋 결과 기록 / iOS·Android 1대씩 동작.
- **WBS**: 4.1.2, 3.2.3, 5.2.2(선택), 6.1, 6.2

### 15주차 — 최종 QA & 🎤 최종 발표
- **목표**: 릴리스 품질 확보 및 최종 시연.
- **산출물**: 데모 빌드(APK), 데모 영상, 최종 발표 자료, 알려진 이슈 목록.
- **검증**: 발표 시나리오 리허설 완주 / 비행기 모드 전 기능 정상 / 신규 기기 설치→첫 사용 흐름 통과.
- **WBS**: 6.2, 6.3, 1.3.2

## 위험 요소 & 대응 방안 (Risk Register)

| # | 위험 | 영향 | 대응 방안 |
|---|---|---|---|
| R1 | **OCR 정확도 미달** — 인쇄물이라도 사진 품질·폰트·레이아웃에 따라 오인식 | 핵심 가치 훼손 | 이미지 전처리(크롭·명암) 도입 / **사용자 텍스트 편집을 항상 fallback으로 보장** / 11·14주차에 측정셋으로 정확도 수치화해 추적 |
| R2 | **일정 압박** — 12주차 중간 발표까지 Must end-to-end 미완 | 발표 데모 실패 | **수직 슬라이스 우선**(완성된 한 흐름 먼저) / Must 동결, Should·Could는 13주차 이후로 / 매주 데모 가능 상태 유지 |
| R3 | **로컬 저장 성능·용량** — 이미지 누적 시 목록·검색 느려지고 용량 증가 | 사용성 저하 | 원본 압축 + 썸네일 분리 저장 / 목록 페이지네이션 / DB 인덱싱(태그·텍스트) / 14주차 100건+ 부하 테스트 |
| R4 | **모델 초기 다운로드 ↔ '100% 오프라인' 충돌** — ML Kit 모델이 첫 실행 시 네트워크 필요 가능 | 비전 원칙 훼손 | 가능 시 모델 **앱에 번들** / 불가 시 첫 실행 1회 다운로드 명시·안내 후 **이후 완전 오프라인 검증**(비행기 모드 회귀 테스트) |
| R5 | **내보내기/가져오기 무결성** — 포맷 불일치·기기 이전 시 데이터 손실 | 데이터 주권 신뢰 붕괴 | 내보내기 포맷에 **버전 필드** 포함 / 가져오기 시 검증·실패 롤백 / 작은 셋부터 round-trip 자동 테스트(13주차) |
