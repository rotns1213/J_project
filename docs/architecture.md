# 오답렌즈 아키텍처 (사람이 읽는 버전)

이 문서는 오답렌즈가 *어떻게 동작하도록 설계됐는지*를 처음 보는 사람이 이해할 수 있게 풀어 쓴 것입니다. 정식 사양과 다이어그램은 `.planning/03-architecture.md`에, 각 결정의 배경은 `.planning/decisions/`의 ADR 문서에 있습니다.

## 한 문장 요약

오답렌즈는 서버가 없습니다. 사진을 찍는 것부터 텍스트를 인식하고, 태그로 분류하고, 나중에 다시 찾는 일까지 **전부 사용자의 폰 안에서** 일어납니다. 인터넷이 끊겨도 동작하고, 학습 데이터는 기기 밖으로 나가지 않습니다.

## 왜 이런 구조인가

오답노트 앱에서 가장 흔한 불편은 두 가지입니다. 사진만 잔뜩 쌓여서 나중에 못 찾는다는 점, 그리고 민감한 학습 기록이 남의 서버로 올라간다는 점입니다. 오답렌즈는 이 둘을 정면으로 해결하려고 "온디바이스 처리 + 로컬 저장"을 기본값으로 못 박았습니다. 그래서 네트워크 통신 코드가 아예 없고, 그만큼 구조가 단순해집니다.

대신 앱 내부는 세 겹으로 나눴습니다. 화면을 그리는 부분, 규칙을 다루는 부분, 실제로 저장하고 글자를 읽어내는 부분입니다. 이렇게 나눈 이유는 나중에 OCR 엔진이나 데이터베이스를 바꾸더라도 화면과 규칙은 건드리지 않게 하기 위해서입니다.

## 세 개의 층

**화면 층**은 사용자가 보는 전부입니다. 촬영·가져오기 화면, 인식된 글자를 고치고 태그를 붙이는 화면, 목록과 검색, 상세 보기, 그리고 내보내기가 있는 설정 화면입니다. 이 층은 "어떻게 저장되는지"는 전혀 모릅니다. 그냥 사용자의 입력을 받아 아래 층에 넘기고, 결과를 보여줄 뿐입니다.

**규칙 층**은 앱의 머리에 해당합니다. "오답을 등록한다", "키워드로 검색한다", "백업 파일을 만든다" 같은 동작의 절차가 여기 있습니다. 중요한 점은 이 층이 ML Kit이나 데이터베이스 같은 구체적인 기술을 **모른다**는 것입니다. 대신 "글자를 읽어주는 무언가", "저장해주는 무언가"라는 약속(인터페이스)에만 기댑니다. 덕분에 테스트할 때 진짜 카메라나 DB 없이 가짜 부품을 끼워 검증할 수 있습니다.

**데이터 층**이 실제 일꾼입니다. 여기서 ML Kit으로 사진의 글자를 읽고, 로컬 데이터베이스(Isar)와 파일 저장소에 이미지·텍스트·태그를 보관하며, 백업 파일을 만들고 복원합니다. 규칙 층이 정한 약속을 이 층이 구현합니다.

## 사진 한 장이 검색 가능한 기록이 되기까지

사용자가 사진을 찍거나 고르면, 그 이미지는 곧장 온디바이스 OCR로 넘어가 글자가 됩니다. 이 과정은 네트워크를 전혀 쓰지 않습니다. 인쇄물이라도 사진 상태에 따라 오인식이 생길 수 있으므로, 사용자가 결과를 직접 고칠 수 있게 편집 화면을 항상 거치도록 했습니다. 여기서 해시태그를 붙이고 저장하면, 원본 이미지는 압축해서, 목록용 작은 썸네일은 따로 보관합니다. 목록 화면은 무거운 원본 대신 썸네일만 불러오기 때문에 기록이 수백 건 쌓여도 느려지지 않습니다.

나중에 다시 찾을 때는 두 갈래입니다. 태그를 골라 같은 분류를 모아 보거나, 인식된 텍스트에서 키워드로 검색하거나. 둘 다 기기 안에서 즉시 처리되므로 지하철처럼 신호가 약한 곳에서도 그대로 동작합니다.

## 데이터를 잃지 않게 하는 법

서버가 없다는 건 자동 백업도 없다는 뜻입니다. 그래서 폰을 바꿀 때를 대비해 "내보내기"를 둡니다. 이미지와 텍스트, 태그를 하나의 압축 파일로 묶어 사용자가 직접 보관하고, 새 기기에서 "가져오기"로 복원합니다. 이때 파일에 형식 버전을 함께 적어 두어, 나중에 앱이 바뀌어도 옛 백업을 안전하게 읽을 수 있도록 했습니다.

## 의도적으로 넣지 않은 것

클라우드 동기화, 계정·로그인, 서버 기반 AI 채점, 수학 수식(LaTeX) 인식은 이번 버전에 넣지 않았습니다. 앞의 셋은 "로컬·오프라인"이라는 핵심 가치와 충돌하고, 수식 인식은 온디바이스로는 난이도가 급격히 올라가기 때문입니다. 수식이 든 문제는 이미지와 태그로 보관하는 방식으로 우회합니다.
